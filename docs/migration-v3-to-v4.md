# Azure Functions Programming Model V3 to V4 Migration Guide

## Overview

This guide documents the migration of the `merchant-func` app from Azure Functions Programming Model V3 to V4. This migration is **required** for Node.js 22+ compatibility, as V3 is not supported on Node 22.

**Migration completed**: February 2026  
**App**: `apps/merchant-func`  
**Functions migrated**: 2 (ValidateOtp, Info)  
**Azure deployment validated**: February 2026

## Why Migrate?

- **Node.js 22+ Requirement**: V3 programming model is incompatible with Node.js 22
- **Modern ESM Support**: V4 uses ES Modules natively, aligning with modern JavaScript
- **Simplified Code**: Eliminates function.json files in favor of code-based configuration
- **Better TypeScript Support**: First-class TypeScript support without adapters
- **Native HTTP Handling**: No need for Express adapters, cleaner HTTP request/response model

## Key Changes Summary

| Aspect                  | V3                               | V4                            |
| ----------------------- | -------------------------------- | ----------------------------- |
| **SDK Package**         | `@azure/functions@^3.x` (devDep) | `@azure/functions@^4.x` (dep) |
| **Module System**       | CommonJS                         | ES Modules (ESM)              |
| **Function Definition** | `function.json` files            | Code-based with decorators    |
| **HTTP Framework**      | Express via adapter              | Native or Express built-in    |
| **Entry Point**         | Individual function exports      | Central `src/functions.ts`    |
| **Context**             | `Context` from V3                | `InvocationContext`           |
| **HTTP Request**        | Express `Request`                | `HttpRequest`                 |
| **HTTP Response**       | Express `Response`               | `HttpResponseInit`            |
| **Middleware**          | Express middleware               | Custom V4 middleware          |

### 2. Update TypeScript Configuration

**tsconfig.json changes:**

```json
{
  "compilerOptions": {
    "target": "es2022", // Was: "es6"
    "module": "nodenext", // Was: "commonjs" - use nodenext for best ESM support
    "moduleResolution": "node16", // Was: "node" - node16 required for nodenext
    "esModuleInterop": true, // Added (this implies allowSyntheticDefaultImports)
    "outDir": "dist",
    "strict": true,
    "skipLibCheck": true,
    "resolveJsonModule": true,
    "sourceMap": true
  }
}
```

**Important:** Using `"module": "nodenext"` and `"moduleResolution": "node16"` provides the most accurate ESM support and ensures TypeScript generates correct import statements.

### 3. Integrate ts2esm for Generated Files

**Critical for Azure deployment:** Generated OpenAPI code must have correct ESM imports with `.js` extensions. The `ts2esm` tool automatically fixes imports in generated files.

**Install ts2esm:**

```bash
pnpm add -D ts2esm@^2.2.7
```

**Update build script in package.json:**

```json
{
  "scripts": {
    "build": "pnpm clean && pnpm generate:definitions && pnpm esm:generated && tsc",
    "esm:generated": "ts2esm tsconfig.json"
  }
}
```

**What ts2esm does:**

- Scans all generated TypeScript files (in `generated/` directory)
- Adds `.js` extensions to relative imports
- Fixes external package imports to use correct ESM paths (e.g., `@pagopa/ts-commons/lib/strings.js`)
- Runs **after** code generation but **before** TypeScript compilation

**Example transformation:**

```typescript
// Before ts2esm (generated by openapi-codegen-ts)
import { PatternString } from "@pagopa/ts-commons/lib/strings";

// After ts2esm
import { PatternString } from "@pagopa/ts-commons/lib/strings.js";
```

This ensures generated models work correctly in Azure Functions V4 runtime without manual import fixes.

### 6. Create V4 Middleware System

Since we removed Express, we need V4-compatible middleware for body validation and error handling.

**New file: `utils/middleware.ts`**

Key exports:

- `HttpResponseInit` - V4 response type
- `toHttpResponse()` - Convert io-functions-commons responses to V4 format
- `requireBodyPayload()` - Body validation middleware using io-ts
- `withMiddlewares()` - Compose middleware chains
- `wrapV4RequestHandler()` - Wrap TaskEither handlers
- `simpleHandler()` - For handlers without body validation

Example usage:

```typescript
import {
  requireBodyPayload,
  withMiddlewares,
  wrapV4RequestHandler,
} from "../utils/middleware.js";

const handler = (
  payload: MyPayload,
  request: HttpRequest,
  context: InvocationContext,
) => pipe();
// business logic returning TaskEither

const middlewareWrap = withMiddlewares(requireBodyPayload(MyPayloadCodec))(
  handler,
);

export default wrapV4RequestHandler(middlewareWrap);
```

### 7. Migrate Function Handlers

#### Pattern: From V3 to V4

**V3 Handler Pattern:**

```typescript
// handler.ts (V3)
import { Context } from "@azure/functions";
import { ContextMiddleware } from "@pagopa/io-functions-commons/...";
import { RequiredBodyPayloadMiddleware } from "@pagopa/io-functions-commons/...";
import * as express from "express";

type Handler = (context: Context, payload: Payload) => Promise<Response>;

export const MyHandler: Handler = async (context, payload) => {
  return pipe(
    // logic
    TE.toUnion,
  )();
};

export const MyFunction = (): express.RequestHandler => {
  const handler = MyHandler();
  const middlewaresWrap = withRequestMiddlewares(
    ContextMiddleware(),
    RequiredBodyPayloadMiddleware(PayloadCodec),
  );
  return wrapRequestHandler(middlewaresWrap(handler));
};
```

**V4 Handler Pattern:**

```typescript
// handler.ts (V4)
import { HttpRequest, InvocationContext } from "@azure/functions";
import {
  requireBodyPayload,
  withMiddlewares,
  wrapV4RequestHandler,
} from "../utils/middleware.js";

type Handler = (
  payload: Payload,
  request: HttpRequest,
  context: InvocationContext,
) => TE.TaskEither<ResponseTypes, IResponseSuccessJson<Result>>;

export const MyHandler: Handler = (payload, _request, _context) => {
  return pipe();
  // logic - NO TE.toUnion, NO ()
};

export const MyFunction = () => {
  const handler = MyHandler();
  const middlewareWrap = withMiddlewares(requireBodyPayload(PayloadCodec))(
    handler,
  );
  return wrapV4RequestHandler(middlewareWrap);
};
```

**Key Changes:**

1. Handler now receives `(payload, request, context)` instead of `(context, payload)`
2. Handler returns `TaskEither` directly, not `Promise` (no `TE.toUnion()`)
3. Import from V4 SDK and new middleware system
4. Add `.js` extensions to local imports (ESM requirement)

#### Pattern: Simple Handlers (No Body Validation)

**V3:**

```typescript
export const InfoHandler = (): Promise<Response> =>
  pipe(healthCheck, TE.toUnion)();

export const Info = (): express.RequestHandler =>
  wrapRequestHandler(InfoHandler());
```

**V4:**

```typescript
export const InfoHandler = (
  _request: HttpRequest,
  _context: InvocationContext,
) =>
  pipe(
    healthCheck,
    // No TE.toUnion
  );

export const Info = () => simpleHandler(InfoHandler());
```

### 8. Migrate Function Index Files

**V3 index.ts:**

```typescript
import { Context } from "@azure/functions";
import createAzureFunctionHandler from "@pagopa/express-azure-functions/...";
import { secureExpressApp } from "@pagopa/io-functions-commons/...";
import * as express from "express";

const app = express();
secureExpressApp(app);

app.post("/api/v1/...", MyFunction());

const azureFunctionHandler = createAzureFunctionHandler(app);

const httpStart = (context: Context): void => {
  setAppContext(app, context);
  azureFunctionHandler(context);
};

export default httpStart;
```

**V4 index.ts:**

```typescript
import { HttpRequest, InvocationContext } from "@azure/functions";
import { HttpResponseInit } from "../../utils/middleware.js";
import { MyFunction } from "./handler.js";

const myFunctionHandler = MyFunction();

export async function myFunction(
  request: HttpRequest,
  context: InvocationContext,
): Promise<HttpResponseInit> {
  context.log(`MyFunction HTTP trigger processed request.`);
  return myFunctionHandler(request, context);
}
```

#### Test Mocks

**V3:**

```typescript
const contextMock = {} as any;

const handler = MyHandler();
const response = await handler(contextMock, payload);
expect(response.kind).toBe("IResponseSuccessJson");
```

**V4:**

```typescript
import { InvocationContext } from "@azure/functions";

const mockContext = {
  log: jest.fn(),
} as unknown as InvocationContext;

const mockRequest = {} as any;

const handler = MyHandler();
const resultTE = handler(payload, mockRequest, mockContext);
const response = await resultTE();

expect(response._tag).toBe("Right");
if (response._tag === "Right") {
  expect(response.right.kind).toBe("IResponseSuccessJson");
}
```

**Key Changes:**

1. Handler signature changed: `(context, payload)` → `(payload, request, context)`
2. Handler returns `TaskEither` - must call `resultTE()` to execute
3. Check `response._tag` ("Left" or "Right") instead of `response.kind`
4. Access success via `response.right`, error via `response.left`

## Common Gotchas

### 1. ESM Import Extensions

**Problem:** TypeScript errors about missing imports, or runtime errors in Azure.

**Solution:** Add `.js` extensions to **ALL imports** - both relative and external packages:

```typescript
// ✅ Correct - relative imports need .js
import { config } from "../utils/config.js";
import { handler } from "./handler.js";

// ✅ Correct - external packages ALSO need .js extensions
import * as TE from "fp-ts/lib/TaskEither.js";
import { pipe } from "fp-ts/lib/function.js";
import { FiscalCode } from "@pagopa/ts-commons/lib/strings.js";
import { IntegerFromString } from "@pagopa/ts-commons/lib/numbers.js";

// ❌ Wrong - missing .js extensions
import { config } from "../utils/config";
import * as TE from "fp-ts/lib/TaskEither";
import { FiscalCode } from "@pagopa/ts-commons/lib/strings";
```

**Critical:** With `"module": "nodenext"` in tsconfig.json, all imports must include `.js` extensions for proper ESM resolution in Azure Functions runtime.

**Automated fix:** Use `ts2esm` to automatically fix imports in generated files (see step 3).

### 2. Handler Return Types

**Problem:** `Type 'Promise<...>' is not assignable to type 'TaskEither<...>'`

**Solution:** Remove `TE.toUnion()` and `()` from handler - return TaskEither directly:

```typescript
// ❌ Wrong (V3 pattern)
return pipe(logic, TE.toUnion)();

// ✅ Correct (V4 pattern)
return pipe(logic);
```

### 3. Test Assertions

**Problem:** `expect(response.kind)` fails - property doesn't exist.

**Solution:** TaskEither uses `_tag` ("Left"/"Right"), not `kind`:

```typescript
// ❌ Wrong
expect(response.kind).toBe("IResponseSuccessJson");

// ✅ Correct
expect(response._tag).toBe("Right");
if (response._tag === "Right") {
  expect(response.right.kind).toBe("IResponseSuccessJson");
}
```

### 4. Function Discovery

**Problem:** `func functions list` shows no functions.

**Solution:** Ensure:

1. `package.json` has `"main": "dist/src/functions.js"`
2. `src/functions.ts` imports and registers all functions
3. Build completed successfully: `pnpm build`

### 5. Module Resolution

**Problem:** Runtime errors in Azure: `Cannot find module '...'` or `ERR_MODULE_NOT_FOUND`.

**Solution:** Ensure correct module configuration and imports:

- All imports must include `.js` extensions (relative AND external packages)
- `tsconfig.json` must use:
  - `"module": "nodenext"` (not "ES2022" or "commonjs")
  - `"moduleResolution": "node16"` (required for nodenext)
- Ensure fp-ts imports use full paths: `fp-ts/lib/Either.js` not `fp-ts/Either`
- Run `pnpm esm:generated` after generating OpenAPI code to fix generated imports

**Debugging tips:**

- Check built files in `dist/` - they should have `.js` in import statements
- Verify `package.json` has `"type": "module"`
- Look for import errors in Azure Application Insights logs

## Azure Deployment Lessons Learned

After successfully deploying to Azure, these critical issues were discovered and resolved:

### Import Extensions Required Everywhere

**Issue:** Functions worked locally but failed in Azure with `ERR_MODULE_NOT_FOUND` errors.

**Root cause:** Local Node.js is more forgiving about missing `.js` extensions in ESM imports. Azure Functions runtime (Node 22) strictly requires them.

**Solution:**

- ALL imports must have `.js` extensions - both relative AND external packages
- Use `"module": "nodenext"` in tsconfig.json (not "ES2022")
- Integrate `ts2esm` to automatically fix generated code
- Verify compiled output in `dist/` has correct import paths

### Generated Code Import Issues

**Issue:** Hand-written code worked, but generated OpenAPI models failed to import.

**Root cause:** `@pagopa/openapi-codegen-ts` generates imports without `.js` extensions.

**Solution:** Add `ts2esm` post-processing step:

```json
{
  "scripts": {
    "build": "pnpm clean && pnpm generate:definitions && pnpm esm:generated && tsc"
  }
}
```

This automatically fixes imports in all generated files before TypeScript compilation.

### Application Insights Initialization

**Issue:** Functions crashed on startup with telemetry errors in Azure (worked locally).

**Root cause:** `ai.defaultClient` may not exist when Application Insights connection string is missing or initialization fails.

**Solution:** Guard the assignment with a null check:

```typescript
export const initTelemetryClient = () => {
  if (!telemetryClient) {
    ai.setup().start();
    if (ai.defaultClient) {
      // Add this check
      telemetryClient = ai.defaultClient;
      telemetryClient.config.samplingPercentage = samplingPercentage;
    }
  }
};
```

### Testing Strategy

**Local testing is not sufficient** - ESM module resolution behaves differently in Azure:

1. ✅ Test locally with `pnpm start`
2. ✅ Run unit tests with `pnpm test`
3. ✅ **Deploy to dev/staging environment**
4. ✅ Test actual endpoints in Azure
5. ✅ Check Application Insights for runtime errors
6. ✅ Verify all functions are discovered and callable

### Build Verification

Before deploying to Azure, verify:

```bash
# Build and check compiled output
pnpm build

# Check imports in compiled files have .js extensions
grep -r "from.*\.js" dist/
grep -r "import.*\.js" dist/

# Should see .js in ALL import statements
```

If any imports are missing `.js`, investigate:

- Are you using `"module": "nodenext"` in tsconfig.json?
- Did `ts2esm` run successfully in the build?
- Are there hand-written imports missing `.js`?
